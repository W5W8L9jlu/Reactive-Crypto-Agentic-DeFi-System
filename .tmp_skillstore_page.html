正在使用“cli-design-expert”。 Design a CLI command for creating feature branches with a help system and error handling

预期结果:

  • Command: branch create <name> [--base <branch>] [--no-PR]
  • Exit codes: 0=success, 1=error, 2=invalid usage, 66=no-input
  • Help shows: usage, options, examples, and common issues
  • Error for missing branch name shows suggestion: 'Run with --base develop'
  • Dangerous actions like force delete require --force flag or confirmation prompt

正在使用“cli-design-expert”。 How should I design exit codes for my CLI tool

预期结果:

  • Use standard exit codes: 0 for success, 1 for general errors
  • Use 2 for invalid usage or arguments
  • Use 66 when input file is not found
  • Use 78 for configuration errors
  • Document all exit codes in --help output

预期结果:

  • Show what file was looking for and where it searched
  • Provide a concrete fix command
  • Link to documentation for more help
  • Use color to highlight the error and the solution

你能构建什么

构建用户友好的CLI工具

创建遵循git和npm等 established 模式的命令，包含清晰的帮助和错误处理。

提升内部工具的采用率

设计开发者愿意使用的CLI，包含渐进式披露和优秀的文档。

完善项目命令界面

为项目工具添加专业的润色，使用语义化颜色、表格和有用的错误建议。

试试这些提示

基本命令设计 
为[你的工具]设计一个CLI命令结构。它应该[描述功能]。遵循CLI最佳实践。
帮助系统 
为名为[name]的命令创建三层帮助系统，包含子命令[列出子命令]。在每个级别包含示例。
错误消息 
为验证[输入类型]的CLI设计错误处理。显示错误代码、消息和常见失败的用户建议。
交互式提示 
为[任务]创建交互式向导流程，对危险操作使用确认。显示inquirer.js集成的代码模式。

最佳实践

  • 遵循名词-动词模式（git commit, npm install）以获得直观的命令结构
  • 为所有常用标志提供短格式（-h）和长格式（--help）选项
  • 设计的错误消息要解释出了什么问题以及如何解决

避免

  • 使用没有解释或建议的神秘错误代码
  • 要求用户阅读外部文档才能进行基本使用
  • 在没有确认或强制标志的情况下执行破坏性操作

常见问题

Does this skill work with all programming languages?
概念普遍适用。代码示例使用TypeScript/Node.js，但模式适用于任何语言。
What exit codes should my CLI use?
标准代码：0表示成功，1表示一般错误，2表示无效用法，66表示找不到输入文件，78表示配置错误。
How do I support shell completion?
为bash、zsh和fish生成补全脚本，解析命令结构并建议子命令。
Is my data safe with this skill?
是的。本技能是只读文档。它提供指导而不访问任何文件或执行代码。
Why are my error messages ignored by users?
用户会忽略没有解决方案的错误。始终包含以'如何修复：'或'运行：'开头的建议行。
How is this different from using commander.js or oclif directly?
这些是框架。本技能提供适用于任何框架或语言的设计原则。
