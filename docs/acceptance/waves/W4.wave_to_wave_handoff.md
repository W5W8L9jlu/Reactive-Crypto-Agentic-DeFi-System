## Frozen capabilities from this wave

- `emergency_force_close` 在合约实现层已落地（`ReactiveInvestmentCompiler.sol`）：
  - `owner()`
  - `setEmergencyAuthorizedRelayer(address relayer, bool authorized)`
  - `isEmergencyAuthorizedRelayer(address relayer)`
  - `emergencyForceClose(bytes32 intentId, uint256 maxSlippageBps) returns (uint256 emergencyExitMinOut)`
- `emergencyForceClose` 的代码级硬约束可见且稳定于当前快照：
  - 仅 owner / authorized relayer 可调用
  - 仅 `ActivePosition` 可调用
  - `maxSlippageBps` 范围 `0..10000`
  - 先写 `Closed` 再输出 `emergencyExitMinOut` 与事件
  - `Closed` 后普通 `executeReactiveTrigger` 会 `ClosedIntentCannotExecute`
- `shadow_monitor` 最小守护闭环已在模块层落地：
  - `ShadowMonitor.reconcile_positions(...) -> ShadowMonitorResult`
  - `ReconciliationDaemon.run_cycle() -> ReconciliationDaemonCycle`
  - `ReconciliationDaemon.run_forever(...) -> list[ReconciliationDaemonCycle]`
  - 模型约束包括：`ActivePosition` 输入、timezone-aware 时间、Grace Period 升级为 critical、输出 `force_close_recommendations`
- 以上冻结仅指“模块接口与代码路径层”，不等同于跨线程端到端冻结。

## Safe truths for next wave to rely on
