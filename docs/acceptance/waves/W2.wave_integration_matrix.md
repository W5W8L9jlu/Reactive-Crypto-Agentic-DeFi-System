## Cross-Thread Interface Notes

### Field naming
- `decision_context_builder` currently relies on normalized fetcher field names such as `confidence`, `depth_2pct`, `tvl`, `daily_trades_executed`, `daily_volume_usd`; concrete provider implementations have not yet proven that they emit those normalized payloads.
- `execution_compiler` freezes contract-facing aliases in camelCase (`intentId`, `inputToken`, `outputToken`, `plannedEntrySize`, `entryAmountOutMinimum`, `entryValidUntil`, `maxGasPriceGwei`, `stopLossSlippageBps`, `takeProfitSlippageBps`, `exitMinOutFloor`), while Python-side typed access remains snake_case. This alias boundary is verified in compiler, validation, and export tests.
- `pre_registration_check` does not currently accept `DecisionContext` or `ExecutionPlan`; its interface remains a separate `RPCStateSnapshot` bundle with no checked adapter from other W2 modules.
- `approval_flow` stable path uses typed `RegisterPayload` fields, but `backend/cli/models.py` still contains an older mapping-style implementation that expects `.get("entryValidUntil")` / `.get("maxGasPriceGwei")`. That is an unresolved internal cross-thread mismatch.

### Enum usage
- `decision_context_builder` uses `TrendDirection = up | down | sideways | unknown`; no downstream W2 consumer of that enum was found.
- `approval_flow` uses `ApprovalAction = approve | reject` and `RiskLevel = low | medium | high`; these remain local to the approval layer and no external CLI route/adaptor handoff was verified.
- No W2 wave relation in the current workspace required a checked enum bridge across modules beyond those local models.

### TTL / timestamp units
- `decision_context_builder.strategy_constraints.ttl_seconds` is integer seconds.
- `execution_compiler` tightens TTL in seconds (`trade_intent.ttl_seconds - CompilerConfig.ttl_buffer_seconds`) and then converts it to Unix timestamp seconds in `register_payload.entry_valid_until`.
- `pre_registration_check` also applies a separate integer `ttl_buffer_seconds` against `trade_intent.ttl_seconds` and returns `remaining_ttl_seconds`.
- `approval_flow` converts `register_payload.entry_valid_until` into `datetime`, but the displayed TTL countdown comes from `DecisionMeta.created_at + DecisionMeta.ttl_seconds`, not from the compiler's tightened expiry. This is a real semantic mismatch, not just an untested seam.

### BPS / pct units
- `execution_compiler`, `validation_engine`, and `approval_flow` consistently treat `max_slippage_bps`, `stop_loss_bps`, `take_profit_bps`, `stopLossSlippageBps`, and `takeProfitSlippageBps` as integer bps.
- `approval_flow` renders bps as percent strings via `bps / 100`, so `100 -> 1.0%`, `300 -> 3.0%`, `800 -> 8.0%`.
- `pre_registration_check` computes `observed_slippage_bps` as a `Decimal`, not an integer. No checked downstream consumer proves how that `Decimal` bps value should be compared, rounded, or handed into compiler/approval paths.
- No wave-level seam was verified for any `pct`-named storage field; current percent behavior is display-only.
