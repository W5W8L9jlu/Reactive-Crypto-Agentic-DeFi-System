## Cross-Thread Interface Notes

### Field naming
- Concrete provider implementations and `decision_context_builder` do not share the same operation vocabulary or payload key set. The builder side expects normalized operations such as `market_trend` and payload keys such as `confidence`, `depth_2pct`, `tvl`; the checked provider implementations expose raw RPC/Graph/Etherscan surfaces instead.
- `execution_compiler` and `reactive_runtime` mostly align on snake_case registration fields, but the compiler model includes `compiled_at` and `compiler_version` that are absent from `ReactiveExecutionPlan`; direct pass-through is therefore `not verified yet`.
- `cli_surface` approval mapping currently expects camelCase register payload keys (`entryValidUntil`, `maxGasPriceGwei`, `entryAmountOutMinimum`) and a mapping-style `.get(...)`, while current compiler/reactive models are snake_case typed objects. This is an unresolved cross-thread mismatch.

### Enum usage
- `ReactivePositionState = PendingEntry | ActivePosition | Closed` is semantically aligned with the Solidity `PositionState` enum names in `IReactiveInvestmentCompiler.sol`.
- `ReactiveTriggerKind = entry | stop_loss | take_profit` has no checked mapping into execution-layer trigger semantics. `RuntimeTrigger` has no `kind` field; it instead requires `trigger_source` plus free-form `metadata`.
- `ExportArtifactKind = machine_truth_json | audit_markdown | investment_memo` is frozen in the CLI layer, but no checked CLI-to-export runtime call was found.

### TTL / timestamp units
- `DecisionContextBuilder` and its handoff use `ttl_seconds` and `timeframe_minutes` as integers; no downstream consumer was found that proves those units remain consistent outside module tests.
- `execution_compiler` / `reactive_runtime` registration payloads use `entry_valid_until` as Unix timestamp integer seconds.
- `execution_layer` requires timezone-aware UTC datetimes for `triggered_at` and `executed_at`.
- `reactive_runtime` does not emit a trigger timestamp at all, so the runtime-to-execution seam currently has a concrete timestamp-shape gap.
- `cli_surface` approval mapping converts `entryValidUntil` from Unix timestamp to `datetime`, but falls back to `DecisionMeta.created_at + ttl_seconds` if that key is absent. Because the expected key name is already mismatched, this timestamp path is not actually verified.

### BPS / percent units
- `TradeIntent`, `Validation.ExecutionHardConstraints`, compiler hard constraints, and reactive hard constraints all keep `*_bps` as integers.
- `cli_surface` formats bps to percent strings via `bps / 100`, so `50 -> 0.5%`, `200 -> 2.0%`, `500 -> 5.0%`.
- `ReactiveRuntime` preserves `stop_loss_slippage_bps` and `take_profit_slippage_bps` as integer bps, but there is no checked bridge proving how those fields relate to execution-layer runtime checks or receipt validation.
- No checked `pct`-typed field was found in the verified wave-level relations; current percentage presentation exists only in CLI display code.
