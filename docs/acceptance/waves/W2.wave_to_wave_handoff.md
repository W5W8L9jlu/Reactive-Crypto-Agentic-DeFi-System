## 2. Safe truths for next wave to rely on

- W2 did **not** freeze an integrated end-to-end flow. What is safe to rely on is limited to module-local boundaries and the explicitly checked seams below.
- `decision_context_builder` safely freezes the normalized `DecisionContext` object shape and its unit conventions, but only for consumers that accept the normalized contract rather than raw provider payloads.
- `execution_layer` safely freezes the minimal `RuntimeTrigger -> ExecutionRecord` transformation boundary, including explicit gate failure on `onchain_checks_passed=False`.
- `reactive_runtime` safely freezes how a registered plan is adapted into the state-machine intent shape:
  - `entry_amount_out_minimum -> entry_min_out`
  - `exit_min_out_floor -> exit_min_out_floor`
  - `planned_entry_size` is preserved as integer size
- `reactive_runtime` safely freezes callback/state rules at the Python port layer:
  - no `entry` in `ActivePosition`
  - no exit trigger in `PendingEntry`
  - no callbacks in `Closed`
- `execution_layer -> export_outputs` has one real checked seam in this wave:
  - `ExecutionRecord.model_dump(mode="json")` is accepted by `export_outputs(...)`
  - this is test-level compatibility, not production integration
- `cli_surface` safely freezes route names and adapter ownership boundaries:
  - downstream work may rely on the nine route names existing
  - downstream work may rely on CLI owning only routing, presentation, and manual entrypoints
  - downstream work may rely on CLI failing explicitly when adapters or runtime dependencies are absent
- Core invariants remain safe to rely on in next-wave planning:
  - machine truth remains structured JSON
  - audit markdown and investment memo remain derived artifacts, not execution truth
  - execution remains conditional/triggered, not free-form market execution
  - reactive runtime remains event-driven and non-decisioning
